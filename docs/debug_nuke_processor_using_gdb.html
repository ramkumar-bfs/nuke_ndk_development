Debug Nuke Plugin with GDB

Beginner-friendly, end-to-end workflow for debugging C++ NDK plugins in VS Code on Linux.

1. What You Are Debugging

A Nuke NDK plugin is a shared library (.so) loaded by the Nuke application at runtime. That means you do not debug a standalone executable for the plugin itself.

Key idea: GDB must debug the Nuke process, then break inside your plugin methods (for example engine(), _validate()).

2. Prerequisites Checklist

  • Nuke is installed and you know the absolute path of the Nuke executable.
  • GDB is installed: gdb --version
  • NDK files exist under: nuke_ndk/linux/<NUKE_VERSION>/include and .../lib
  • Build task creates debug symbols (-DCMAKE_BUILD_TYPE=Debug).
In this workspace, debug build output is expected under: build-debug/plugin/Linux/<NUKE_VERSION>/

3. Debug Architecture in This Workspace

Component Purpose
.vscode/tasks.json Builds plugin with debug symbols.
.vscode/launch.json Launches or attaches GDB to Nuke process.
.gdbinit Sets pending breakpoints and useful GDB defaults.
NUKE_PATH Tells Nuke where to load your built plugin from.

4. Build the Plugin in Debug Mode

Use VS Code task or run command manually.

Option A: VS Code Task

  1. Open Command Palette.
  2. Run task: Build Nuke Plugin (Debug Symbols).

Option B: Terminal Command

cd /path/to/your/workspace
export NUKE_NDK_ROOT="$PWD/nuke_ndk"
export NUKE_VERSION="16.1v2"
cmake -S . -B build-debug -DNUKE_NDK_ROOT="$NUKE_NDK_ROOT" -DNUKE_VERSION="$NUKE_VERSION" -DCMAKE_BUILD_TYPE=Debug
cmake --build build-debug --target package_plugin -j

Verify Symbols

file build-debug/plugin/Linux/16.1v2/PassThroughLite.so

You should see text similar to with debug_info, not stripped.

5. Launch Debug Flow (Recommended for Beginners)

  1. Set breakpoints in plugin code (engine(), _validate(), etc.).
  2. Open VS Code Run and Debug.
  3. Select Nuke: Launch with GDB (Debug Plugin).
  4. Enter Nuke executable path when prompted (example: /opt/Nuke16.1v2/Nuke16.1).
  5. Wait for Nuke to open under debugger.
  6. Create or run the node in Nuke so plugin code executes.
If breakpoint circles look unresolved at startup, that is normal. Your plugin symbols bind after the shared library is loaded.

6. Attach Debug Flow (When Nuke Is Already Open)

  1. Start Nuke manually first.
  2. In VS Code, choose Nuke: Attach GDB to Running Process.
  3. Pick the Nuke process ID.
  4. Trigger your node execution in Nuke.
  5. GDB stops at plugin breakpoints when reached.

8. Typical First Debug Session

  1. Put a breakpoint at the first line of engine().
  2. Launch with GDB config.
  3. In Nuke, add your node to the graph and force evaluation.
  4. When paused, inspect values (y, x, r, channels).
  5. Step over one scanline copy loop and watch row data changes.
  6. Continue until you confirm expected frame behavior.

9. Troubleshooting

Breakpoint never hits

  • Confirm plugin loaded from debug output path in NUKE_PATH.
  • Confirm node is actually used in current script graph.
  • Rebuild debug binary and verify with debug_info.

Symbols missing or optimized behavior

  • Make sure build type is Debug.
  • Delete old build dir if needed and reconfigure.

GDB safe-path warning for .gdbinit

This is normal security behavior. Optional fix:

echo 'add-auto-load-safe-path /absolute/path/to/workspace/.gdbinit' >> ~/.gdbinit

Nuke starts but plugin does not appear

  • Verify plugin path exists and has .so file.
  • Verify node registration string and menu path in plugin source.

10. Practical Tips for Beginners

  • Start by debugging one method only (usually engine()).
  • Use small test images for faster iteration.
  • Inspect call stack whenever behavior is surprising.
  • Prefer launch mode first; use attach mode later.
  • Keep one known-good Nuke script for repeatable tests.

11. Quick Start Summary

  1. Build debug plugin.
  2. Run Nuke: Launch with GDB (Debug Plugin).
  3. Add/use node in Nuke.
  4. Hit breakpoint and inspect state.
  5. Step and verify pixel logic.
