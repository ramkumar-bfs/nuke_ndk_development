Why the Plugin Failed on Another System

A beginner-friendly explanation of the root cause, diagnosis steps, and the fix.

1. Short Answer First

Your plugin was built with machine-specific information, so when moved to another computer it still tried to use your original paths.

Main runtime issue: the plugin contained an absolute library search path (RPATH) pointing to your local machine.
Secondary debug issue: debug symbols included your source folder path. This usually does not break runtime, but it can confuse debugging on another system.

2. What Exactly Was Causing the Issue

A. Runtime Portability Problem (Critical)

The built plugin shared object had an embedded RPATH like: /home/pipeline/development/nuke_ndk_development/nuke_ndk/linux/16.1v2/lib.

On another machine, that exact folder does not exist. So the loader may fail to resolve dependent libraries correctly.

B. Source Path in Debug Symbols (Non-critical for runtime)

Debug information (DWARF) stored your original source path. On another machine, debugger may try to open files from that old location.

This usually affects debugging experience, not plugin loading itself.

3. How the Issue Was Identified

Two inspections were used:

Step 1: Check dynamic section of the plugin

readelf -d build-debug/plugin/Linux/16.1v2/PassThroughLite.so

Result showed an absolute RPATH entry to your local path, confirming portability risk.

Step 2: Check debug information paths

readelf --debug-dump=info build-debug/plugin/Linux/16.1v2/PassThroughLite.so

Result showed DW_AT_name and DW_AT_comp_dir with local absolute directories.

Diagnosis flow:
  1. Plugin works locally but fails remotely.
  2. Inspect runtime metadata for hardcoded paths.
  3. Find absolute RPATH in shared object.
  4. Inspect DWARF to explain source-path references.
  5. Apply CMake portability fix and rebuild.
  6. Re-verify metadata after rebuild.

4. Technical Concepts in Simple Words

Term Simple Meaning Why You Care
Shared object (.so) A plugin binary loaded by another app (Nuke). This is your deployable artifact.
RPATH Built-in hint telling Linux where to search libraries. If absolute and machine-specific, portability breaks.
DWARF debug info Metadata for debugger (files, lines, variables). Can reference old source paths if not remapped.
Prefix-map flags Compiler options to rewrite path prefixes inside debug info. Prevents leaking local paths and improves portability.

5. What Was Changed to Fix It

In CMake, two key improvements were added:

  1. Disable RPATH embedding for portable plugin binaries.
  2. Remap debug source paths to relative paths using compiler prefix-map flags.
# Keep plugin binaries portable across machines by avoiding absolute RPATHs.
set(CMAKE_SKIP_RPATH ON)

option(SANITIZE_DEBUG_PATHS "Remap source paths in debug info to relative paths" ON)
if(SANITIZE_DEBUG_PATHS AND CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
  add_compile_options(
    "-ffile-prefix-map=${CMAKE_SOURCE_DIR}=."
    "-fdebug-prefix-map=${CMAKE_SOURCE_DIR}=."
  )
endif()
After rebuild, dynamic section no longer had RPATH/RUNPATH, and debug paths changed to relative values like ./src/PassThroughLite.cpp.

6. Why This Works

  • No hardcoded local library path means the plugin is not tied to one machine layout.
  • Relative debug paths avoid debugger trying to open files from old absolute directories.
  • You can now move the plugin package to another compatible system more safely.

7. What Still Must Match on Target System

Even with path fixes, binary compatibility still matters.
  • Target machine must have compatible Nuke version.
  • Linux distribution and glibc/libstdc++ compatibility must match enough for your build.
  • Required runtime libraries (for example Nuke libraries) must be available.
  • NUKE_PATH must include the folder containing your plugin .so.

8. Quick Verification Commands

On build machine

# 1) Confirm there is no RPATH/RUNPATH
readelf -d build-debug/plugin/Linux/16.1v2/PassThroughLite.so | grep -E 'RPATH|RUNPATH|NEEDED'

# 2) Confirm debug path mapping looks relative
readelf --debug-dump=info build-debug/plugin/Linux/16.1v2/PassThroughLite.so | grep -E 'DW_AT_name|DW_AT_comp_dir' | head

# 3) Confirm debug symbols exist
file build-debug/plugin/Linux/16.1v2/PassThroughLite.so

On target machine

# Optional dependency check
ldd /path/to/PassThroughLite.so

# Then run Nuke with plugin path available
export NUKE_PATH="/path/to/plugin/folder:${NUKE_PATH}"

9. Beginner Mental Model

  1. Runtime loading depends on library resolution and ABI compatibility.
  2. Debugger behavior depends on symbol info and source path mapping.
  3. Fix runtime first, then improve debug convenience.
If plugin fails on another machine, first inspect runtime metadata (RPATH, needed libs), not only source-path references in debug symbols.
